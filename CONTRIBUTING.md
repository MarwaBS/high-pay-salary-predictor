# Contributing to High-Paying Jobs in the US

Thank you for considering a contribution. This document describes the workflow
and standards for keeping the project consistent and reproducible.

## Setup

```bash
git clone https://github.com/MarwaBS/High_pay_Analysis_us.git
cd High_pay_Analysis_us
make install          # creates .venv and installs all dependencies
pre-commit install    # installs git hooks (ruff, nbstripout, file hygiene)
```

## Development workflow

1. **Create a branch** from `main`:
   ```bash
   git checkout -b feat/your-feature-name
   ```

2. **Make your changes.** Run the quality checks before committing:
   ```bash
   make lint          # ruff check
   make format        # ruff format (auto-fixes style)
   make type-check    # mypy static analysis
   make test          # full pytest suite
   ```

3. **Commit.** Pre-commit hooks run automatically:
   - `ruff` lints and auto-fixes imports/style
   - `ruff-format` normalizes formatting (Black-compatible)
   - `nbstripout` strips notebook outputs so diffs stay readable
   - General file hygiene (trailing whitespace, merge markers, large files)

4. **Open a pull request** against `main`. The CI pipeline runs lint + tests
   on Python 3.10 and 3.11 automatically.

## Code conventions

| Convention | Detail |
|------------|--------|
| **Line length** | 120 characters (configured in `pyproject.toml`) |
| **Formatter** | `ruff format` (Black-compatible, run with `make format`) |
| **Imports** | `ruff check --select I` sorts them automatically |
| **Type hints** | Required for all new public functions |
| **Shared code** | Feature constants and `engineer_features` live in `pipeline.py` — do not duplicate them |
| **Config** | All thresholds and paths belong in `config.yaml`, never hardcoded |

## Adding or changing features

- Update `FEATURES_FULL` in `pipeline.py` if the model feature set changes.
- Retrain the model with `make model` (quantile trainer) and verify tests still pass.
- The regression guard is `tests/test_pipeline.py::test_saved_metrics_within_expected_range`. If quantile coverage, crossings, or P50 R² drift, update the bands there deliberately and explain why in the commit message.

## Adding tests

- Unit tests go in `tests/test_pipeline.py` (data, features, ML) or `tests/test_api.py` (API endpoints).
- Session-scoped fixtures (`cfg`, `df`, `df_engineered`) are in `tests/conftest.py`.
- Aim to keep the test suite green: `make test` must pass before opening a PR.

## Data

- **Do not commit large data files.** The pre-commit hook blocks new files > 5 MB.
- Raw source data lives in `Resources/`. The cleaned dataset in `Data/` is regenerated by notebook 1.
- If you update the cleaning logic, re-run `make data` and commit the updated CSV.

> **Known tech debt:** `Resources/census_data.csv` (~20 MB) and
> `us_state/us_state.shp` (~15 MB) were committed before the 5 MB pre-commit
> gate was added, so they still live in git history and inflate `.git` to
> ~100 MB. A `git filter-repo` rewrite is tracked as a future follow-up.
> Do not add new large binaries — use object storage instead.

## Notebooks

- Notebook outputs are stripped automatically by `nbstripout` on commit — regenerate them locally.
- Keep the four-notebook structure (`cleaning → EDA → mapping → ML`).
- If you add a new notebook, prefix it numerically (`05_...`) and add it to the pipeline table in the README.
